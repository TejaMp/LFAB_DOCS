## 1. **Exact Commands for the Whole Operation**

### **A. Data Pump Export (Backup)**

bash
expdp system/password DIRECTORY=INSISHIST_DIR \
  DUMPFILE=INSISHIST_%U.dmp LOGFILE=INSISHIST_export.log \
  TABLESPACES=INSIS_HIST_DATA PARALLEL=10

- **Exports** all data from the INSIS_HIST_DATA tablespace into dump files.

---

### **B. Drop Disks from ASM Diskgroup**


ALTER DISKGROUP DATA DROP DISK DATA29 REBALANCE POWER 1000;
ALTER DISKGROUP DATA DROP DISK DATA28 REBALANCE POWER 1000;

- **Removes** disks DATA29 and DATA28 from the ASM diskgroup named DATA.
- REBALANCE POWER 1000 speeds up the rebalance (data redistribution).

---

### **C. Add Disks to Another Diskgroup**


ALTER DISKGROUP ARCHIVE ADD DISK 'AFD:DATA29' REBALANCE POWER 1000;
ALTER DISKGROUP ARCHIVE ADD DISK 'AFD:DATA28' REBALANCE POWER 1000;

- **Adds** the disks (now named as AFD:DATA29 and AFD:DATA28) to the ARCHIVE diskgroup.

---

### **D. Create New Tablespace**


CREATE BIGFILE TABLESPACE INSIS_HIST_DATA
  DATAFILE '+DATA' SIZE 1700G AUTOEXTEND ON NEXT 1G
  ENCRYPTION USING 'AES256'
  DEFAULT STORAGE(ENCRYPT);

- **Creates** a new bigfile tablespace in the +DATA ASM diskgroup.

---

### **E. Data Pump Import (Restore)**

bash
impdp system/password DIRECTORY=INSISHIST_DIR \
  DUMPFILE=INSISHIST_%U.dmp LOGFILE=INSISHIST_import.log \
  TABLESPACES=INSIS_HIST_DATA PARALLEL=10

- **Imports** all data back into the new tablespace.

---

### **F. Monitor ASM Rebalance**


SELECT * FROM v$asm_operation;

- **Monitors** the progress of disk rebalance operations.

## 2. **What is AFD?**

- **AFD** stands for **ASM Filter Driver**.
- It is an Oracle technology that provides I/O filtering and validation for ASM disks at the OS level.
- When you see AFD:DATA28, it means the disk is managed by ASM Filter Driver and is named DATA28.

**Example:**
- 'AFD:DATA28' is the device name as seen by Oracle ASM, not the OS path (like /dev/sdX).

---

## 3. **How to Know if Disks Can Be Deleted?**

- **Check if the disks are in use:**  
  You should only drop a disk from an ASM diskgroup if:
  - There is enough free/usable space in the diskgroup **after** dropping.
  - No datafiles (tablespaces) are solely residing on those disks.

**Check disk usage:**

SELECT name, path, total_mb, free_mb, state
FROM v$asm_disk
WHERE header_status = 'MEMBER' AND group_number = (SELECT group_number FROM v$asm_diskgroup WHERE name = 'DATA');


- **Check rebalance:**  
  After you issue the DROP DISK command, ASM will **rebalance** the data, moving it off those disks before actually removing them.

---

## 4. **How to Check Which Disks Hold Tablespace Data?**

- **Find datafiles for tablespace:**

SELECT file_name
FROM dba_data_files
WHERE tablespace_name = 'INSIS_HIST_DATA';

- The file_name will look like +DATA/.... (ASM path).

- **Find which ASM disks hold a file:**

-- Get the file number from the ASM file name
SELECT file_number, name
FROM v$asm_file
WHERE group_number = (SELECT group_number FROM v$asm_diskgroup WHERE name = 'DATA');

-- Then, for a specific file, see which disks it spans
SELECT a.name AS disk_name, b.file_number, b.incarnation, b.block_number
FROM v$asm_disk a, v$asm_file b
WHERE a.group_number = b.group_number
  AND b.file_number = <your_file_number>;

- **Note:** In ASM, files are striped and mirrored across multiple disks, so a tablespace is usually spread across many disks, not just two.

---

## 5. **How to Know If Tablespace Data Is Only on Those Two Disks?**

- **In ASM, this is unlikely!**  
  ASM stripes data across all disks in the diskgroup for redundancy and performance.
- If you drop two disks, ASM will **move data off them** (rebalance) before completing the drop, so you don’t lose data.
